# Интеграции личного дашборда

Все настройки находятся на экране **«Интеграции»** и сохраняются вместе с остальными данными. Экспорт в Obsidian и файловый мост Codex выполняются локально; платный OpenAI API не нужен.

## Obsidian

1. Нажмите **«Выбрать»** и укажите корневую папку хранилища — ту, внутри которой лежит каталог `.obsidian`.
2. При желании измените папку назначения внутри хранилища.
3. Нажмите **«Проверить»**.
4. Нажмите **«Экспортировать»**. Каждая заметка дашборда будет записана отдельным `.md`-файлом.

Повторный экспорт обновляет файл, пока название заметки не менялось. YAML-свойства можно отключить переключателем; старые файлы дашборд сам не удаляет.

Каждая новая запись из «Записать и осмыслить» автоматически появляется среди обычных заметок с тегом `осмысление` и входит в следующий экспорт. Исходный текст сохраняется без изменений. Удаление записи из истории не удаляет уже созданную заметку. Экспорт обрабатывает весь переданный список заметок в пределах общего лимита размера локального запроса, без скрытого ограничения по количеству файлов.

Принятые смысл и вопрос из нового разбора можно отдельной кнопкой дописать в связанную локальную заметку. Приложение не перезаписывает её существующее содержимое. Это действие само по себе не обращается к Obsidian: дополнение попадёт в `.md`-файл только после следующего ручного нажатия **«Экспортировать»**. Последующее изменение решения или удаление исходной записи не редактирует уже дописанный фрагмент и не удаляет внешний файл.

## Мост Codex

1. Выберите категории, которые разрешено включить в снимок, и нажмите **«Опубликовать снимок»**. Безопасная проекция появится в `bridge/dashboard-snapshot.json`.
2. Codex может прочитать снимок и подготовить действия в `bridge/codex-commands.json` по формату из `bridge/codex-commands.example.json`.
3. Нажмите **«Проверить команды»**.
4. Просмотрите очередь и нажмите **«Применить разрешённые»**.

Разрешения на создание, изменение и завершение задач, а также на работу с заметками задаются отдельными переключателями. Подтверждённые действия сохраняются в `bridge/archive/`.

Снимок версии 2 не содержит настройки, пути интеграций, журнал действий, личный контекст или память помощника. Личные записи осмысления входят в него только при отдельном явном включении категории `journal`, которая по умолчанию выключена. Связанные с размышлениями заметки также исключаются, даже если включена категория «Заметки»; внутренний признак происхождения продолжает защищать их после удаления исходной записи или изменения тега. В Obsidian они при этом экспортируются как обычно. Служебный текст задач, проектов и событий также не передаётся. Ранее опубликованный снимок можно удалить из моста отдельной кнопкой.

### Разбор записи через Codex

Для осмысления используется отдельный канал, не связанный с командами изменения данных:

1. Дашборд показывает точный текст и предлагает отдельно отметить нужные разделы личного контекста и до шести активных формулировок памяти.
2. Выбранная память показывается дословно; по умолчанию она и личный контекст пусты.
3. После подтверждения создаётся `bridge/reflection-analysis-request.json` версии 3; если ничего дополнительного не выбрано, в нём только запись.
4. Codex готовит `bridge/reflection-analysis-response.json` по примеру в каталоге моста и копирует контрольную сумму запроса.
5. Дашборд проверяет идентификаторы, версию исходного текста и контрольную сумму всего точного пакета, включая выбранную память.
6. После фактического локального сохранения оба чувствительных файла удаляются без переноса в архив.
7. Пользователь подтверждает, исправляет или отклоняет сохранённый локальный черновик.

Версия bridge остаётся 3, а ответ сохраняет полный объект `analysis`. Для новых ответов интерфейс локально и без дополнительного обращения к модели создаёт до трёх необязательных предложений: `meaning` берётся из `possibleExplanation`, `question` — из одноимённого поля, `next_action` — из `proposedAction`. Каждое предложение редактируется отдельно и имеет состояние «ожидает решения», «принято» или «отклонено». Общая проверка анализа — подтверждение, поправка или исключение — остаётся независимой.

Само принятие предложения не создаёт и не изменяет никаких связанных сущностей. Только отдельная кнопка дописывает принятые `meaning` и `question` в локальную reflection-заметку. Другая отдельная кнопка превращает принятый `next_action` в незапланированную задачу во «Входящих»; задача не попадает автоматически в план и календарь. Созданные связи сохраняются, но изменение решения или удаление исходной записи не переписывает заметку, задачу или внешний файл. Никакие предложения не становятся памятью, не меняют профиль и не влияют на планировщик автоматически. Анализы из резервных копий v10 остаются без решений; они не достраиваются задним числом при переходе на v11 или текущую v13.

`LifeArea[]`, описания сфер и устойчивые `areaId` не публикуются в общем снимке Codex. Если пользователь разрешил категорию «Проекты», схема снимка версии 2 сохраняет прежнее совместимое поле `area` только как название контекста проекта. Добавление сфер жизни не расширяет уже выданные разрешения.

Одновременно активен только один запрос, срок его жизни — 24 часа. Пока он не завершён, дашборд не перезапишет его другой записью; повторная отправка идентичного запроса безопасна, а другой пакет с тем же ID отклоняется. Ожидание можно отменить, не удаляя исходный текст. Пауза или удаление элемента памяти запрещает его будущий выбор, но не отзывает уже подтверждённый одноразовый запрос; его можно отменить отдельно. Ни один результат анализа сам по себе не создаёт задачи и не меняет данные пользователя. Мост является локальным файловым обменом: он не вызывает модель автоматически и не требует платного API.

Следующий технический долг — будущий контракт bridge v4 с контрольной суммой ответа и идемпотентным подтверждением очистки после сохранения. В текущей версии 3 эти свойства не заявлены как реализованные.

## Google

Контур данных уже подготовлен: календарь, задачи, интервал проверки, отдельный календарь фокус-блоков, односторонний или двусторонний режим и политика разрешения конфликтов. Эти правила можно настроить сейчас.

Фактический вход в Google и OAuth пока намеренно не активированы. Когда будет удобно выполнить подключение, потребуется добавить авторизацию и сетевой транспорт; структура данных и интерфейс переделки не потребуют.
