# ADR-001. Вычисляемые wiki-ссылки и сохраняемые предметные отношения

Дата: 22 июля 2026 года
Статус: принято

## Контекст

Документы ПСОЖ имеют устойчивые `DocumentId`, но пока физически представлены `Note`, native-объектами и материалами. Пользователь записывает внутренние ссылки и встраивания непосредственно в каноническом тексте как `[[Название]]` и `![[Название]]`.

Экспериментальный срез сохранял отдельную relation с binding metadata для каждого wiki-token. Такая модель переживала переименование, но создавала вторую скрытую интерпретацию текста: локальная база отдельно запоминала target, позицию, occurrence и fingerprint. После свободного редактирования Markdown текст и bindings требовали reconciliation, собственной миграции, корзины и recovery. Это противоречит принципу единственного канонического пользовательского текста и усложняет будущую файловую совместимость.

При этом `objectGraph.relations` уже нужен для самостоятельных предметных отношений: структурного `contains` и явно созданных смысловых `links`/`embeds`, которые существуют независимо от конкретного текстового token.

## Решение

- Объект сохраняет устойчивую идентичность. Несколько представлений одного объекта не создают копии его данных.
- Канонический текст документа является единственным источником wiki-синтаксиса.
- `documentWikiLinks.ts` строит полностью пересоздаваемый reference index из актуальных `DocumentRecord`, их текста, заголовков и ID. Индекс не записывается в `DashboardState`, не требует миграции, корзины или pending recovery.
- Backlinks, переходы и документные read-only embeds вычисляются из этого индекса.
- `objectGraph.relations` хранит только самостоятельные предметные `contains`, `links` и `embeds`. Эти relations проходят единый application boundary, реальную проверку mixed endpoints, общие инварианты, удаление, восстановление и pending recovery.
- `ObjectRelation` не содержит wiki origin или binding metadata.
- Ровно один документ с нормализованным заголовком даёт `resolved`; отсутствие цели — `unresolved`; несколько целей — `ambiguous`; ссылка на исходный документ — `self`.
- Для одинаковых заголовков chooser не обещает скрытую ID-привязку и не вставляет token. Пользователь должен переименовать один из документов.
- Переименование строит чистый план по состоянию до изменения title. Переписываются только распознанные tokens, которые однозначно разрешались именно в переименовываемый `DocumentId` и находятся в редактируемом plain-text source. Каждый source обновляется через канонический `DocumentRepository`; read-only, structured, ambiguous и unresolved sources не упрощаются и отражаются в summary.
- `![[...]]` читает актуальный target без копирования контента. Рекурсивное отображение ограничено глубиной 3 и множеством посещённых `DocumentId`.
- До появления файловой иерархии не вводятся aliases, пути, скрытые UUID или proprietary wiki-синтаксис. Адреса вида `[[Люди/Александр]]` относятся к следующему этапу.

## Совместимость данных

Схема остаётся v16 ради `pendingRelations` и relations в снимках корзины Note/native-object.

- v15 relations мигрируют как plain semantic relations; dangling edges переходят в pending recovery.
- Экспериментальная v16 распознаётся по `origin`, `binding` или wiki-specific pending reason и получает отдельную неперезаписываемую safety-копию.
- `origin: "manual"` преобразуется в plain relation.
- `wiki-link` и `wiki-embed` удаляются из live graph и trash snapshots, поскольку заново выводятся из текста.
- Wiki-specific pending entries удаляются; manual pending entries сохраняются как plain relations.
- Пользовательский текст при преобразовании не изменяется. Неинтерпретируемая semantic relation останавливает загрузку fail-closed.

## Отклонённые альтернативы

- Persistent relation на каждый token: создаёт скрытую вторую интерпретацию Markdown и сложный lifecycle вычисляемых данных.
- Скрытый ID в тексте: ухудшает переносимость и вводит собственный синтаксис до решения файловых путей.
- Автоматический выбор одного из одноимённых документов: создаёт непредсказуемую ссылку.
- Глобальная замена старого title: может изменить обычный текст, ambiguous и unrelated references.
- Немедленная физическая миграция всех документов в `UniversalObject`: выходит за утверждённую гибридную архитектуру.

## Последствия и ограничения

- Reference index можно полностью удалить и восстановить без потери пользовательских данных.
- Удаление wiki-token немедленно убирает backlink; удаление target делает token unresolved; восстановление target снова разрешает его по актуальному title.
- Без путей одинаковые titles честно остаются ambiguous.
- Если read-only или structured source нельзя переписать при rename, старая метка может стать unresolved. Скрытый alias не создаётся.
- Иерархия, пути, aliases, Markdown-файлы, вложения и двусторонняя Obsidian-синхронизация остаются задачами этапа 2.3.
