# ADR: граница Composite Widget, Widget Blocks и Document Blocks

Статус: принято как архитектурная граница; конкретная реализация редакторов и каталога блоков ещё не утверждена.  
Дата: 23 июля 2026 года.  
Связанный аудит: [`../WIDGET_FOUNDATION_DONOR_AUDIT.md`](../WIDGET_FOUNDATION_DONOR_AUDIT.md).

## Контекст

ПСОЖ развивает два внешне похожих механизма:

1. универсальные документы с будущим мультимодальным содержимым;
2. составные виджеты, позволяющие собирать на холсте несколько представлений и layout containers в один экземпляр.

Без явной границы эти механизмы могут превратиться в две несовместимые блочные системы, каждая из которых хранит текст, объекты и вложенность по собственным правилам. Это создаст:

- второй редактируемый источник пользовательских данных;
- расхождение документов и их копий внутри widget config;
- дублирование editor, renderer и migration logic;
- неясность владения задачами, проектами, документами и событиями;
- невозможность безопасно переиспользовать представления в документах и на холсте.

## Решение

### 1. Предметные данные принадлежат доменным модулям

Документы, задачи, проекты, события, люди, материалы и другие предметные сущности имеют устойчивую идентичность и канонический источник данных в соответствующем домене.

`WidgetInstance`, `CompositeWidgetConfig` и `WidgetBlockInstance` не владеют независимыми копиями этих сущностей.

### 2. Document Blocks описывают содержание документа

Document Blocks относятся к document domain. Они могут представлять:

- текст;
- заголовки;
- списки;
- таблицы;
- изображения;
- аудио и видео;
- файлы;
- ссылки;
- встраивания существующих объектов;
- другие элементы содержимого документа.

Точный persisted format Document Blocks, Markdown representation и правила синхронизации утверждаются отдельно в документной архитектуре.

Эта ADR не фиксирует окончательный формат Document Blocks.

### 3. Widget Blocks описывают композицию представления

Widget Blocks относятся к presentation/composition layer. Они используются внутри одного Composite Widget для:

- размещения нескольких представлений;
- группировки;
- вертикальной или горизонтальной композиции;
- секций;
- колонок;
- небольших подписей и пояснений;
- показа объекта по binding;
- показа сохранённой выборки;
- наследования контекста пространства;
- настройки действий над каноническими данными.

Widget Block не становится документом, задачей, проектом, событием или новым универсальным объектом.

### 4. Один внешний контракт виджета

Native и Composite Widgets используют общий внешний `WidgetInstance`:

```text
Space
  → Canvas
      → WidgetInstance
          ├── native config
          └── composite config → Widget Blocks
```

Composite Widget не создаёт отдельную параллельную систему размещения на уровне Canvas. Для Canvas он остаётся одним перемещаемым и изменяемым по размеру экземпляром виджета.

### 5. Binding вместо копирования

Widget Block, который показывает предметные данные, использует binding:

- `object`;
- `saved-query`;
- `space-context`;
- `external`;
- ограниченный `static`;
- либо `inherit` от родительского Composite Widget.

Для предметных данных binding хранит идентификатор или правило выборки. Он не хранит независимую редактируемую копию объекта.

### 6. Ограничение `static` content

`static` content допускается для небольшого содержимого композиции:

- заголовка;
- подписи;
- короткого пояснения;
- разделителя;
- простой ссылки;
- декоративного элемента.

Он не предназначен для хранения:

- длинной заметки;
- статьи;
- дневниковой записи;
- самостоятельного знания;
- полноценного документа;
- копии задачи, проекта или события.

Когда статическое содержимое становится самостоятельным пользовательским материалом, UI должен предложить создать или связать документ.

### 7. Общие renderer-компоненты допустимы и желательны

Один renderer предметного объекта может использоваться:

- в native widget;
- внутри Composite Widget;
- во встраивании в документ;
- на странице проекта;
- в поиске или сохранённом представлении.

Переиспользование renderer не означает объединение persisted форматов Document Blocks и Widget Blocks.

### 8. Действия направляются в доменные команды

Действие внутри Widget Block или native widget не изменяет локальную копию данных. Оно вызывает domain/application command:

- завершить задачу;
- изменить срок;
- обновить документ;
- создать объект;
- перенести событие;
- добавить связь.

Persisted widget config может хранить настройки доступности или отображения действий, но не сериализованные исполняемые функции.

## Ограничения вложенности

Устанавливаются два независимых ограничения.

### Loader safety limit

Loader обязан безопасно прочитать внешнее, неизвестное или повреждённое дерево и остановить чрезмерную рекурсию. Технический предел donor prototype — 32 уровня и 1000 узлов.

Это аварийная защита данных, а не продуктовая возможность редактора.

### Editor creation policy

Редактор первой версии разрешает не более двух уровней вложенности layout containers внутри Composite Widget.

Редактор не должен создавать структуру, близкую к техническому loader limit.

## Не принято этим решением

Эта ADR не определяет:

- окончательный формат Document Blocks;
- конкретный Markdown AST;
- layout engine холста;
- конкретные React renderers;
- полный каталог Widget Blocks;
- визуальный editor Composite Widget;
- совместное редактирование;
- пользовательский JavaScript;
- физическую модель хранения всех предметных сущностей.

Эти вопросы требуют отдельных решений и прототипов.

## Запрещённые реализации

Запрещается:

1. хранить полный текст канонического документа в Widget Block config как независимую редактируемую копию;
2. хранить копию задачи, проекта или события вместо binding;
3. превращать Widget Blocks во второй object graph;
4. давать Widget Block самостоятельный предметный жизненный цикл без отдельного решения;
5. связывать семантические отношения предметных объектов преимущественно с block instance;
6. использовать Composite Widget как альтернативное хранилище документов;
7. считать технический предел глубины разрешённой глубиной редактора;
8. создавать отдельный Canvas layout item для каждого внутреннего блока Composite Widget.

## Следствия для реализации

Перед созданием реальных block definitions необходимо:

- иметь безопасные `WidgetBlockDefinitionRegistry`, loader и serializer;
- поддерживать unknown block preservation;
- поддерживать binding inheritance;
- обеспечить supported parent with unsupported child;
- валидировать child policy;
- отделить persisted instance от runtime load result;
- описать UI policy перевода большого static text в документ;
- использовать domain commands для изменения предметных данных.

Первыми допустимыми blocks после интеграции фундамента могут быть:

- `section`;
- `stack`;
- `columns`;
- короткий `text-label`;
- `object-view`;
- `saved-query-view`.

Их появление не означает автоматического принятия полноценного блочного редактора.

## Критерии соблюдения ADR

Решение считается соблюдённым, если:

- удаление Composite Widget не удаляет показанные предметные объекты;
- редактирование объекта из виджета изменяет его каноническую версию;
- один объект может отображаться в нескольких blocks и widgets без копирования;
- Widget Block config остаётся конфигурацией представления;
- существенный пользовательский текст хранится в document domain;
- document embeds и widgets могут использовать общие renderers и domain commands;
- loader сохраняет неизвестные blocks без потери;
- редактор не создаёт более двух уровней layout containers в первой версии.
