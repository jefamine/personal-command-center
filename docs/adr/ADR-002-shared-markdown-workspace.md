# ADR-002. Общее Markdown-пространство ПСОЖ и Obsidian

Дата: 22 июля 2026 года
Статус: принято

## Контекст

ПСОЖ уже имеет единый прикладной контракт документов, но их текущие канонические данные находятся в `DashboardState`: обычные документы представлены `Note`, нативные документы — объектным графом, материалы — `ReadingItem`. Ручной Markdown-экспорт создаёт отдельный файл-снимок и не позволяет ПСОЖ и Obsidian редактировать одно документное пространство.

Пользователю нужен выбор: оставить внутреннее IndexedDB-хранилище либо явно подключить весь Obsidian vault, его подпапку или любую другую папку Markdown. Во втором случае оба приложения должны работать с одними физическими файлами без скрытого зеркала текста в локальной базе.

Браузерный File System Access API асинхронен, требует явного разрешения и доступен не на всех платформах. Поэтому файловый доступ нельзя помещать в React-компоненты или считать браузерный API окончательной архитектурой настольного приложения.

## Решение

### Два режима, один прикладной слой

- `internal` сохраняет текущий IndexedDB-контур, существующие данные, миграции, корзину и поведение `DocumentRepository`.
- `external-folder` использует явно выбранную папку как каноническое хранилище документного текста.
- В файловом режиме Markdown-файл является текущей версией документа; ПСОЖ и Obsidian являются редакторами одних файлов.
- Единая workspace-сессия предоставляет React вычисляемый каталог и прикладные команды. Компоненты не вызывают `showDirectoryPicker`, File System Access API или IndexedDB напрямую и не выбирают поведение по физическому источнику.
- `DocumentRepository` эволюционирует в единую прикладную границу двух режимов. Синхронное чтение возможно из актуального снимка сессии; операции файлового режима асинхронны и возвращают типизированные ошибки и конфликты.

### Канонические и служебные данные

- В `DashboardState` не сохраняется независимо редактируемая полная копия внешнего Markdown-текста.
- Отдельное локальное хранилище может содержать directory handle, переносимый descriptor подключения, индекс путей, хеши, mapping устойчивых ID, состояние интерфейса, историю операций и ограниченные защитные snapshots. Эти данные не становятся текущим текстом документа.
- Directory handle сохраняется отдельно и не входит в JSON-экспорт. После переноса копии или потери разрешения папку нужно выбрать повторно.
- Внутри выбранной папки после первой подтверждённой записи допустима служебная `.psozh/`: `workspace.json`, `trash/`, `conflicts/`. Простое сканирование её не создаёт.
- `.obsidian`, `.psozh`, `.git` и временные файлы редакторов полностью исключаются из документного scanner. ПСОЖ никогда не изменяет `.obsidian`.

### Идентичность

- Новый или явно переведённый под управление ПСОЖ файл получает `psozh-id` в frontmatter.
- Для managed-файла `DocumentId` строится из `psozh-id`; rename и move не меняют идентичность.
- Повторный `psozh-id` является конфликтом. Файлы не объединяются и не исправляются автоматически.
- Существующий файл без `psozh-id` читается, редактируется и участвует в wiki-ссылках с временной path-based identity. Сканирование не модифицирует его.
- Долговечная semantic relation к unmanaged-файлу запрещена до отдельного явного действия «Сделать документ управляемым ПСОЖ».
- Это действие перечитывает файл, проверяет ожидаемый хеш и frontmatter и минимально добавляет ID без изменения body и неизвестных metadata.

### Frontmatter и текст

- Читаются `psozh-id`, `tags` и `aliases`; остальные свойства допустимы.
- Обычное изменение body повторно использует исходный raw frontmatter байт-в-байт и не прогоняет его через сериализацию.
- Точечные metadata-изменения используют консервативное минимальное редактирование. Неизвестные свойства, комментарии, порядок и пользовательское форматирование сохраняются.
- Malformed или неподдерживаемый для безопасной правки frontmatter остаётся читаемым, но операции изменения metadata блокируются с понятной ошибкой.

### Пути, ссылки и отношения

- Внутренние пути нормализуются относительно корня workspace с `/`; абсолютные пути и выход через `..` запрещены.
- `[[...]]` и `![[...]]` остаются обычным Obsidian-совместимым синтаксисом. Не вводятся скрытые UUID или сохраняемые token bindings.
- Ссылка разрешается детерминированно: точный путь от корня, путь относительно source, уникальный basename, уникальный alias; иначе результат `ambiguous` или `unresolved`.
- Alias после `|`, heading после `#` и block suffix после `^` сохраняются при безопасном переписывании пути.
- Wiki links и backlinks вычисляются из актуальных файлов. `ObjectRelation` остаётся отдельным сохраняемым предметным графом и не создаётся на каждый Markdown-token.

### Запись, перемещение и конфликты

- Перед записью, rename, move, добавлением ID и восстановлением адаптер перечитывает файл и сравнивает expected content hash.
- Внешнее изменение никогда не перезаписывается молча. Возвращается typed conflict с действиями перечитать, сравнить или сохранить пользовательскую версию отдельной копией.
- Rename и move физически меняют путь. Fallback выполняет verified copy и удаляет исходник только после повторного чтения и проверки нового файла.
- Обновляются только ранее однозначно разрешённые wiki-token в доступных редактируемых source-файлах. Частичный результат всегда сопровождается summary.
- Удаление из ПСОЖ перемещает файл в `.psozh/trash`; conflict copy помещается в `.psozh/conflicts` и получает новый `psozh-id`. Внешнее удаление проводником не имитирует корзину ПСОЖ.

### Подключение и миграция

- Выбор папки возможен только по явному пользовательскому действию с feature detection и проверкой разрешения.
- Первое подключение выполняет только read-only scan и показывает сводку. До подтверждённой записи файлы, ID, ссылки и YAML не меняются.
- Миграция внутренних документов является отдельным двухфазным мастером: dry-run, safety backup, immutable plan, подтверждение, запись, повторное чтение и проверка, manifest, отдельное подтверждение активации.
- `Note`, простой одноблочный native-document и material преобразуются по явно описанной политике; structured-document пропускается без упрощения.
- Существующий `DocumentId` сохраняется как `psozh-id`. Старые внутренние данные не удаляются и служат rollback-контуром, но после активации не показываются второй копией.
- Rollback только возвращает режим `internal` и не удаляет созданные Markdown-файлы.

### Адаптеры

- `InternalDocumentStorageAdapter` сохраняет совместимость внутреннего режима.
- `BrowserDirectoryDocumentStorageAdapter` является первым внешним адаптером.
- `MemoryDocumentStorageAdapter` обеспечивает основные автоматические тесты без обращения к пользовательской файловой системе.
- Контракт допускает будущий desktop adapter с watcher без миграции Markdown-документов.

## Отклонённые альтернативы

- Импорт/экспорт или периодическая двусторонняя синхронизация двух текстовых баз: создаёт конфликтующие канонические копии.
- Полное зеркало vault в IndexedDB: нарушает единственный источник текста и усложняет конфликты.
- Автоматическая разметка всего существующего vault идентификаторами: неожиданно модифицирует пользовательские файлы.
- Сохраняемая relation на каждый wiki-token: дублирует интерпретацию Markdown и противоречит ADR-001.
- Прямой File System API из React: привязывает предметную модель к браузеру и делает тестирование зависимым от личной файловой системы.
- Немедленная миграция `DashboardState` v17 только ради файлового режима: смешивает независимые жизненные циклы и усложняет безопасный rollback.

## Последствия и ограничения

- Файловый режим в браузере зависит от поддержки File System Access API, разрешений вкладки и ручного rescanning; постоянный watcher появится только в desktop adapter.
- File System Access API не предоставляет атомарный compare-and-swap или перенос. Браузерный адаптер повторно проверяет хеш непосредственно в файловом порту, сохраняет операции как verified copy-before-delete и никогда намеренно не выбирает «последняя запись победила», но полностью исключить микрогонку с внешней программой между последней проверкой и системным commit браузера способен только будущий native desktop adapter.
- На телефонах и в неподдерживаемых браузерах внутренний режим остаётся полноценным и честно объясняет ограничение.
- Heading/block navigation, сложный merge, вложения, Canvas, Bases, Dataview, WYSIWYG и межустройственная синхронизация не входят в этот milestone.
- Path-based identity unmanaged-файла меняется при внешнем rename; устойчивая связь требует явного `psozh-id`.
- Ограниченные revision snapshots являются резервными версиями, а не текущим текстом, и не участвуют в обычном чтении.
