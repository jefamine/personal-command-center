## 1. Назначение документа

Этот файл фиксирует, какие части экспериментальной реализации WO1–WO3 считаются полезным донором, какие требуют переработки при переносе, а какие запрещено превращать в постоянную архитектуру ПСОЖ.

Донорский репозиторий имеет независимую Git-историю и не является источником истины проекта. Его код нельзя слепо переносить всей цепочкой коммитов или считать автоматически принятым только потому, что тесты в донорской среде проходили.

После этой фиксации донор считается замороженным на SHA `09043823699285bf6ce1f9134a7d4cd57acd3821`. Новые Work Orders в нём до интеграции в основной репозиторий не начинаются.

## 2. Общий вывод

Донорская реализация содержит полезное доменное ядро:

- пространства и их предметные привязки;
- отдельные холсты;
- экземпляры виджетов;
- типизированные источники данных;
- responsive layouts;
- реестры определений;
- версионирование и runtime validation;
- безопасное сохранение неизвестных схем;
- legacy migration;
- технический прототип составных виджетов и деревьев блоков.

При этом она ещё не является готовой системой пространств и виджетов. В доноре отсутствуют хранение в реальном `DashboardState`, IndexedDB migration, layout engine, React-интеграция, реальные новые виджеты и пользовательский сценарий холста.

Основная архитектурная оценка:

1. WO1 принят концептуально как основа модели.
2. WO2 принят частично: API реестров, migration scenarios и safe-loading pattern полезны, но обычный widget pipeline требуется укрепить.
3. WO3 принят как технический прототип: безопасность дерева и round-trip preservation полезны, но Composite Widget требует отдельной границы с документными блоками.

## 3. Что переносится как принятая основа

### 3.1. `SpaceBinding`

Сохраняются привязки пространства к:

- отсутствующему предметному контексту;
- папке;
- сфере жизни;
- проекту;
- сохранённому представлению.

Привязка пространства задаёт контекст, но не делает пространство владельцем связанных предметных данных.

### 3.2. `Space`

Сохраняется базовая модель:

- устойчивый `id`;
- `schemaVersion`;
- односторонняя ссылка `canvasId`;
- название;
- иконка;
- цвет;
- binding;
- порядок;
- timestamps.

Удаление пространства не должно удалять предметные объекты, связанные с его контекстом.

### 3.3. Минимальный `CanvasDefinition`

Сохраняется отдельная идентичность холста и направление `Space → Canvas`.

Текущий минимальный Canvas не считается окончательной полной схемой. Его нельзя заранее наполнять случайными полями до выбора layout engine, storage model, history model и responsive override policy.

### 3.4. `WidgetDataBinding`

Сохраняются пять канонических источников:

- `object`;
- `saved-query`;
- `space-context`;
- `static`;
- `external`.

Binding хранит способ получения данных, а не независимую копию предметного объекта.

### 3.5. Responsive layouts

Сохраняются:

- `desktop` как обязательная раскладка;
- необязательные `tablet` и `mobile` overrides;
- прямоугольник `x`, `y`, `w`, `h`.

Новая продуктовая модель не связывается с четырьмя legacy-размерами `full`, `two-thirds`, `half`, `third`. Они применяются только при детерминированной миграции старого layout.

### 3.6. `WidgetKind`

Сохраняется различие:

- `native` — специализированный виджет;
- `composite` — составная композиция представлений и layout-blocks.

Оба вида должны использовать общий внешний контракт `WidgetInstance`.

### 3.7. Safe-loading result

Сохраняется различие между:

- поддерживаемым экземпляром;
- неизвестным типом;
- более новой schema version;
- более новой config version;
- повреждённой структурой;
- повреждённой конфигурацией.

Неизвестные и повреждённые данные должны сохраняться как исходный `raw` без молчаливого урезания. Один проблемный элемент не должен прекращать загрузку остальных элементов массива.

### 3.8. Registry creation-result pattern

Сохраняется модель создания реестра через результат:

- `ok: true` с полностью сформированным registry;
- `ok: false` со структурированными ошибками.

Частично созданный registry при наличии ошибок не используется. Снимки определений, массивов bindings/views и constraints должны быть защищены от последующей мутации входных объектов.

### 3.9. Legacy migration scenarios

Сохраняются:

- детерминированная сортировка по legacy `order` и исходному индексу;
- first-fit placement на 12-колоночной сетке только для миграции;
- сохранение старых `gridWidth` и `gridHeight`, когда они валидны;
- fallback из legacy `size`;
- `enabled: false` → hidden instance;
- `reflection` → `document` с предупреждением;
- сохранение неизвестного legacy widget как raw;
- типизированные warnings.

### 3.10. Persisted/runtime split для Widget Blocks

Сохраняется разделение:

- persisted `WidgetBlockInstance` содержит persisted children;
- runtime `LoadedWidgetBlock` содержит `WidgetBlockLoadResult[]`.

Runtime metadata не записывается внутрь persisted-формата.

### 3.11. Безопасность дерева блоков

Сохраняются:

- аварийный loader depth limit;
- общий node-count limit;
- detection циклов;
- глобальная проверка duplicate IDs;
- обход неизвестных descendants;
- безопасный публичный loader;
- private recursive loading;
- сохранение unsupported children внутри supported parent.

Технический loader limit и продуктовый editor limit считаются разными ограничениями.

### 3.12. Raw-preserving serialization

Сохраняется принцип:

- unsupported элемент сериализуется исходным `raw`;
- supported элемент сохраняет неизвестные дополнительные поля;
- канонические изменённые поля накладываются поверх исходного объекта;
- load/serialize не мутирует входные данные.

## 4. Что переносится только с переработкой

### 4.1. `WidgetInstance`

Сохраняется общий каркас:

- identity и versioning;
- `canvasId`;
- `widgetType`;
- visibility;
- binding;
- view;
- actions;
- presentation;
- layouts;
- config;
- timestamps.

До storage integration необходимо уточнить минимальные устойчивые контракты:

- `WidgetViewConfig`;
- `WidgetPresentationConfig`;
- `WidgetBehaviorConfig`;
- декларации доступных действий.

Нельзя оставлять все перечисленные области бесконтрольными `Record<string, unknown>` без границы ответственности. При этом не требуется заранее строить максимальную универсальную схему: часть специфики должна принадлежать конкретному `WidgetDefinition` и UI registry.

### 4.2. Обычный `WidgetDefinitionRegistry`

При переносе требуется довести его до уровня финального block registry:

- настоящий plain-record check по prototype;
- строгая проверка `description`;
- проверка каждого binding against canonical union;
- проверка `migrateConfig`, если он присутствует;
- структурная проверка результата `validateConfig`;
- typed snapshot без wholesale double casts;
- защита от мутации исходных definitions.

### 4.3. Обычный widget loader

Текущий донорский loader нельзя считать готовым. При переносе требуется:

- требовать `configVersion >= 1`;
- строить `WidgetInstance` из проверенных полей, а не приводить весь raw объект к типу;
- валидировать binding как discriminated union;
- структурно валидировать результат `validateConfig`;
- сохранять неизвестные поля supported instance при round trip;
- сохранять unsupported raw по исходной ссылке;
- гарантировать batch isolation;
- не выполнять молчаливое понижение новых schema/config versions.

### 4.4. `actions`, `presentation`, `behavior`

Три уровня действий сохраняются концептуально:

- domain actions;
- view actions;
- instance actions.

Но persisted instance не должен хранить произвольные исполняемые команды. В нём должны храниться настройки доступности и конфигурация представления, а фактические обработчики предоставляет registry/application layer.

`behavior` необходимо явно выделить для таких параметров, как:

- height-by-content;
- internal scroll;
- min/max height behavior;
- collapsed state;
- overflow behavior.

### 4.5. Built-in definitions

Legacy `overview`, `focus`, `plan`, `inbox`, `weather`, `reading`, `recommendations`, `document` и `custom` могут временно переноситься как compatibility layer.

Они не считаются вечными фундаментальными типами новой платформы. Часть из них позднее должна стать:

- шаблоном пространства;
- составным виджетом;
- сохранённым представлением;
- конфигурацией более универсального native widget.

`custom` не должен становиться постоянной catch-all моделью для текста, ссылки, изображения, metric, file и API одновременно.

### 4.6. Legacy migration implementation

При переносе алгоритм сохраняется, но production implementation должна:

- принимать явные target `spaceId` и `canvasId`;
- не использовать non-null assertions для binding/layout;
- получать metadata/defaults из registry, где это возможно;
- проверять созданные instances через новый loader/validator;
- сохранять structured migration report;
- работать внутри реальной storage migration только после резервного snapshot.

## 5. Composite Widget: принятая граница

Composite Widget допускается как композиция представлений и действий.

Widget Blocks:

- не являются предметными объектами;
- не заменяют документы, задачи, проекты, события и другие домены;
- не становятся вторым object graph;
- не владеют копиями предметных данных;
- используют bindings на канонические данные;
- могут содержать layout containers и небольшое статическое оформление.

До создания каталога реальных blocks требуется отдельное ADR о границе между Document Blocks и Widget Blocks.

## 6. Что запрещено переносить как постоянное решение

### 6.1. Независимую историю donor commits

Запрещён blind cherry-pick полной истории донорского репозитория. В ней присутствуют неудачные промежуточные реализации, повторные исправления одних контрактов и временные отклонения от Work Orders.

Перенос выполняется в новой чистой ветке основного репозитория от актуального `main`.

### 6.2. Catch-all `custom` как основу платформы

Legacy compatibility допустима. Постоянный универсальный тип, где одно поле `variant` превращает виджет в заметку, ссылку, изображение, показатель, файл или API, не принимается.

### 6.3. Неограниченные unknown-словари

`unknown` допустим на runtime boundary и внутри generic config конкретного definition. Он не должен подменять устойчивую модель всех actions, presentation, behavior и persisted settings.

### 6.4. Widget Blocks как второй формат документов

Запрещено хранить внутри block config независимые копии полноценных документов, задач, проектов или событий. Полноценный пользовательский текст должен жить в document domain, а block показывает или встраивает его.

### 6.5. Legacy widget types как окончательную продуктовую систему

Старые карточки сохраняются ради перехода и непрерывности пользовательского опыта. Они не ограничивают будущую систему готовыми жёсткими комбинациями источника и представления.

## 7. Граница `static` content

`static` content предназначен для небольших элементов композиции:

- подписи;
- краткого пояснения;
- заголовка;
- разделителя;
- простой ссылки;
- небольшого декоративного содержимого.

Он не предназначен для долговременного хранения статьи, дневниковой записи, большой заметки или иного полноценного документа. UI должен предлагать перевод существенного текста в document domain.

## 8. Два ограничения вложенности

Устанавливаются два разных понятия:

1. Loader safety limit — аварийная защита при чтении внешних, неизвестных или повреждённых данных. Донорское значение 32 может использоваться как технический предел.
2. Editor creation policy — продуктовый предел, который редактор разрешает создать пользователю. Для первого этапа контейнеры ограничиваются двумя уровнями вложенности внутри composite widget.

Loader safety limit не является разрешением редактору создавать 32 уровня контейнеров.

## 9. Порядок интеграции

1. Завершить и безопасно интегрировать документное файловое пространство.
2. Создать новую integration-ветку в основном репозитории от актуального `main`.
3. Перенести принятые контракты и тестовые сценарии из donor SHA.
4. Одновременно переписать обычный widget registry/loader по усиленным runtime boundaries.
5. Зафиксировать ADR о Composite Widget и Document Blocks.
6. Запустить `npm test`, `npm run check`, `npm run build` в реальном основном репозитории.
7. Только после этого проектировать `DashboardState` migration и IndexedDB persistence.
8. Не начинать реальный каталог Widget Blocks до принятия ADR и согласования с документным слоем.

## 10. Критерий завершённого переноса

Донорский фундамент считается интегрированным, когда:

- код находится в основном репозитории на ветке от актуального `main`;
- нет слепого переноса независимой donor history;
- ordinary widgets и Widget Blocks имеют сопоставимые safe-loading guarantees;
- неизвестные схемы проходят lossless round trip;
- legacy migration детерминирована и не удаляет данные;
- Composite Widget не дублирует document domain;
- тесты, typecheck и build проходят в реальном проекте;
- storage и UI ещё не обязаны быть реализованы, но их дальнейшие границы не противоречат фундаменту проекта.
