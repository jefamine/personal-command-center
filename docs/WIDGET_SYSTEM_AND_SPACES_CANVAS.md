# Система виджетов и холст пространств ПСОЖ

Статус: целевая продуктово-архитектурная спецификация.  
Дата фиксации: 22 июля 2026 года.  
Связанные документы: [`ROADMAP.md`](ROADMAP.md), [`ARCHITECTURE.md`](ARCHITECTURE.md), [`PRODUCT_MAP.md`](PRODUCT_MAP.md).

## 1. Назначение документа

Этот документ фиксирует целевую модель верхних вкладок, пространств, виджетов, представлений и редактируемого холста ПСОЖ.

Он отвечает на вопросы:

- что является объектом, а что только способом его показа;
- что такое пространство верхней панели;
- из каких частей состоит виджет;
- какие семейства виджетов нужны системе;
- как пользователь добавляет, перемещает, изменяет и настраивает виджеты прямо на холсте;
- как должна работать свободная, но упорядоченная сетка;
- как один и тот же объект показывается в разных местах без копирования;
- как виджеты, документы и семантические связи должны сосуществовать;
- как сохранить возможность расширения системы без бесконечного роста жёстко прошитых типов.

Эта спецификация предшествует реализации нового layout engine. Сначала утверждается модель системы, затем выбираются технические библиотеки и пишется код.

## 2. Продуктовая формула

> Каждая верхняя вкладка ПСОЖ является самостоятельным настраиваемым пространством с собственным холстом, раскладкой и набором виджетов.

Пространство может быть связано с:

- папкой документов;
- сферой жизни;
- проектом;
- сохранённым представлением;
- либо быть полностью пользовательским.

Пользователь должен редактировать пространство непосредственно там, где он им пользуется:

- создавать вкладки на верхней панели;
- переименовывать их;
- менять порядок, цвет и иконку;
- добавлять виджеты прямо на холст;
- перемещать и растягивать их мышью;
- менять источник данных и способ отображения рядом с самим виджетом;
- удалять представление, не удаляя показанные данные.

Глубокий раздел настроек не должен быть обязательным для обычной настройки пространства.

## 3. Базовые понятия

### 3.1. Объект

Объект — это настоящая предметная сущность системы со своей устойчивой идентичностью и жизненным циклом.

Примеры:

- документ;
- задача;
- проект;
- событие;
- материал;
- сфера жизни;
- человек;
- показатель;
- файл;
- будущие типы предметных сущностей.

Объект существует независимо от конкретного экрана, пространства или виджета.

### 3.2. Представление

Представление — способ показать объект или набор объектов.

Примеры для одного набора задач:

- список;
- таблица;
- доска;
- календарь;
- временная линия;
- компактный счётчик;
- диаграмма.

Представление не создаёт новую копию данных.

### 3.3. Виджет

Виджет — настраиваемое окно, через которое пользователь видит объект, коллекцию, действие, навигацию, аналитику или содержимое.

Виджет не является владельцем показанных данных. Удаление виджета не удаляет документ, задачу, проект, событие или материал.

### 3.4. Экземпляр виджета

Экземпляр виджета — конкретное размещение определённого виджета в определённом пространстве.

Один тип виджета может быть добавлен много раз с разными настройками.

Например:

- список задач сферы «Психиатрия»;
- список задач проекта «Ординатура»;
- компактный список задач на сегодня;
- тот же тип виджета с разными источниками, фильтрами и внешним видом.

### 3.5. Пространство

Пространство — самостоятельный настраиваемый дашборд, доступный как верхняя вкладка или вложенное сохранённое рабочее представление.

Примеры:

- Главная;
- Психиатрия;
- Здоровье;
- Отношения;
- Литература;
- отдельный проект;
- пользовательское пространство без предметной привязки.

### 3.6. Раскладка

Раскладка — расположение и размеры экземпляров виджетов в конкретном пространстве для конкретного класса экрана.

Раскладка является данными интерфейса, а не предметными данными пользователя.

## 4. Главный принцип устройства виджета

Каждый виджет строится из независимых частей:

```text
Источник данных
+
Способ отображения
+
Доступные действия
+
Внешний вид
+
Поведение в пространстве
```

Пример:

```text
Источник: задачи сферы «Психиатрия»
Отображение: компактный список
Действия: создать, завершить, открыть
Внешний вид: без рамки, плотный режим
Поведение: высота по содержимому до установленного максимума
```

Другой экземпляр может использовать тот же источник:

```text
Источник: задачи сферы «Психиатрия»
Отображение: календарь
Действия: открыть, перенести на другой день
Внешний вид: широкая панель с заголовком
Поведение: фиксированная высота и внутренний скролл
```

Система не должна создавать отдельный жёсткий тип для каждой комбинации источника и вида. Она должна собирать виджеты из совместимых источников, представлений, действий и настроек.

## 5. Источники данных

Виджет может получать данные одним из следующих способов.

### 5.1. Один конкретный объект

Примеры:

- конкретный документ;
- конкретная задача;
- конкретный проект;
- конкретное событие;
- конкретный материал.

Связь хранится по устойчивому идентификатору объекта.

### 5.2. Сохранённая выборка

Примеры:

- задачи на сегодня;
- активные проекты сферы;
- документы выбранной папки;
- непрочитанные материалы;
- объекты, связанные с текущим проектом.

Выборка включает фильтры, сортировку, группировку и ограничение результатов.

### 5.3. Контекст пространства

Виджет может автоматически использовать привязку пространства.

Пример: пространство связано со сферой «Психиатрия», а виджет «Задачи» показывает задачи текущей сферы без жёстко записанного идентификатора в каждом шаблоне.

Такой источник должен явно показывать пользователю, что он зависит от контекста пространства.

### 5.4. Статическое содержимое

Примеры:

- произвольный текст;
- изображение;
- ссылка;
- подпись;
- декоративный разделитель;
- инструкция или пояснение.

### 5.5. Внешний источник

Примеры:

- погода;
- Google Calendar;
- API;
- внешний файл;
- будущие интеграции.

Внешний источник должен иметь явные состояния загрузки, ошибки, отсутствия разрешения и устаревания данных.

## 6. Семейства виджетов

### 6.1. Виджеты одного объекта

Показывают один конкретный объект:

- документ;
- задача;
- проект;
- событие;
- материал;
- человек;
- показатель;
- файл.

Возможные режимы:

- компактная карточка;
- подробная карточка;
- встроенный редактор;
- read-only preview;
- показатель состояния.

### 6.2. Виджеты коллекций

Показывают набор объектов:

- задачи;
- документы;
- проекты;
- события;
- материалы;
- связанные объекты;
- обратные ссылки;
- результаты поиска.

Поддерживаемые представления развиваются независимо:

- список;
- таблица;
- доска;
- календарь;
- галерея;
- временная линия;
- группы;
- компактная лента.

### 6.3. Навигационные виджеты

- дерево папок;
- избранное;
- недавно открытые;
- история переходов;
- оглавление документа;
- быстрые ссылки;
- список пространств;
- навигация по проекту или сфере.

### 6.4. Виджеты быстрого ввода и действий

- создать задачу;
- записать мысль;
- добавить событие;
- добавить материал;
- начать таймер;
- отметить показатель;
- открыть командную палитру;
- выполнить пользовательскую команду.

Виджет может быть кнопкой, компактной формой или полноценной панелью ввода.

### 6.5. Аналитические виджеты

- прогресс проекта;
- выполненные задачи;
- нагрузка;
- распределение времени;
- динамика показателя;
- сводка по сфере;
- график;
- число;
- статус;
- будущие объяснимые выводы помощника.

Аналитика должна показывать источник расчёта и не выдавать предположение за факт.

### 6.6. Содержимое и медиа

- текст;
- Markdown-фрагмент;
- изображение;
- галерея;
- аудио;
- видео;
- PDF;
- файл;
- веб-ссылка;
- вложение.

### 6.7. Виджеты отношений

- связанные объекты;
- обратные ссылки;
- родители и дочерние объекты;
- объекты той же сферы;
- граф ближайших связей;
- список отношений выбранного типа.

### 6.8. Контейнеры

- секция;
- вертикальная стопка;
- горизонтальная группа;
- вкладки внутри блока;
- сворачиваемая группа;
- переключаемые представления одного источника.

Вложенность контейнеров должна быть ограничена. Начальное целевое ограничение — не более двух уровней вложенности внутри пространства.

## 7. Реестр виджетов

Каждый тип виджета регистрируется через единый контракт.

Концептуально определение должно включать:

```ts
interface WidgetDefinition {
  type: string;
  version: number;
  title: string;
  description: string;
  family: WidgetFamily;
  supportedBindings: DataBindingKind[];
  supportedViews: string[];
  defaultView: string;
  settingsSchema: WidgetSettingsSchema;
  capabilities: WidgetCapabilities;
  layoutConstraints: WidgetLayoutConstraints;
  createDefaultConfig(context: WidgetCreationContext): WidgetConfig;
  migrateConfig?(fromVersion: number, value: unknown): WidgetConfig;
}
```

Точные имена определяются при реализации, но обязательны следующие свойства:

- стабильный идентификатор типа;
- версия конфигурации;
- схема допустимых настроек;
- поддерживаемые источники данных;
- поддерживаемые представления;
- минимальные и рекомендуемые размеры;
- доступные действия;
- правила миграции старых конфигураций;
- состояния загрузки, ошибки и отсутствия данных;
- возможность проверить конфигурацию до сохранения.

Нельзя хранить все настройки всех виджетов в одном бесконтрольном словаре без типизации и версионирования.

## 8. Модель экземпляра виджета

Концептуально экземпляр должен хранить:

```ts
interface WidgetInstance {
  id: string;
  widgetType: string;
  widgetVersion: number;
  spaceId: string;
  titleOverride: string | null;
  binding: WidgetDataBinding;
  view: WidgetViewConfig;
  actions: WidgetActionConfig;
  presentation: WidgetPresentationConfig;
  behavior: WidgetBehaviorConfig;
  layouts: ResponsiveWidgetLayouts;
  createdAt: string;
  updatedAt: string;
}
```

Экземпляр не содержит копию предметных данных. Он хранит только способ их получения и показа.

## 9. Действия внутри виджетов

Действия делятся на три уровня.

### 9.1. Действия над данными

Примеры:

- завершить задачу;
- изменить срок;
- создать объект;
- отредактировать документ;
- перенести событие;
- добавить связь.

Они вызывают доменные команды, а не изменяют локальную копию внутри виджета.

### 9.2. Действия над представлением

Примеры:

- сменить список на таблицу;
- изменить сортировку;
- свернуть группы;
- показать или скрыть поля;
- изменить диапазон календаря.

### 9.3. Действия над экземпляром виджета

Примеры:

- переместить;
- изменить размер;
- настроить;
- дублировать;
- скопировать;
- заблокировать;
- удалить с холста.

Удаление экземпляра не вызывает удаление источника данных.

## 10. Режимы работы холста

### 10.1. Обычный режим

Пользователь работает с содержимым:

- читает;
- редактирует текст;
- нажимает ссылки;
- завершает задачи;
- взаимодействует с календарём;
- открывает объекты.

Случайное перемещение и изменение размеров виджетов заблокировано.

### 10.2. Режим компоновки

Включается одной заметной командой прямо на холсте:

```text
Настроить пространство
```

После включения:

- отображается сетка и направляющие;
- появляются ручки перемещения и изменения размера;
- становятся доступны добавление, копирование, удаление и группировка;
- содержимое виджетов не перехватывает жесты компоновки;
- можно отменять и повторять изменения.

Выход из режима возвращает обычное взаимодействие с данными.

## 11. Редактирование прямо на холсте

### 11.1. Добавление

На свободной области и в общей панели доступна команда `+`.

Каталог добавления должен поддерживать:

- поиск;
- семейства;
- недавно использованные;
- рекомендуемые для текущего пространства;
- шаблоны;
- предпросмотр;
- добавление нескольких экземпляров одного типа.

После выбора виджет появляется в ожидаемом месте, а не в случайной части холста.

### 11.2. Быстрая панель виджета

При выделении виджета рядом с ним показывается компактная панель:

```text
Переместить · Размер · Настроить · Дублировать · Удалить
```

Основные изменения выполняются без перехода в глобальные настройки.

### 11.3. Инспектор

Настройки открываются рядом с виджетом или в контекстной боковой панели и сразу отражаются на холсте.

Группы настроек:

- источник;
- фильтр;
- сортировка и группировка;
- представление;
- действия;
- заголовок;
- рамка и фон;
- плотность;
- поведение высоты;
- отображение на разных экранах.

### 11.4. Базовые операции

Обязательны:

- drag-and-drop;
- resize за края и углы;
- дублирование;
- копирование и вставка;
- блокировка размещения;
- выделение нескольких элементов;
- выравнивание;
- равные интервалы;
- перенос между пространствами;
- undo/redo;
- клавиатурное удаление и перемещение;
- доступные действия без мыши.

## 12. Сетка

### 12.1. Основной принцип

Сетка должна ощущаться почти свободной, но сохранять геометрический порядок.

Используется мелкая адаптивная система делений с магнитным прилипанием:

- края виджетов совпадают;
- интервалы единообразны;
- элементы не перекрываются случайно;
- соседние размеры предсказуемы;
- свободное место не превращается в хаотические щели.

Не закреплять продуктовую модель за текущими вариантами «треть», «половина», «две трети» и «вся ширина».

### 12.2. Размеры

Пользователь свободно меняет:

- ширину;
- высоту;
- минимальную и максимальную высоту;
- режим высоты по содержимому;
- внутренний скролл;
- свёрнутое и развёрнутое состояние.

Ограничения конкретного виджета задаются его определением и объясняются пользователю.

### 12.3. Форма

Область размещения виджета в основной сетке остаётся прямоугольной ради устойчивой компоновки и адаптивности.

Визуальное содержимое внутри может выглядеть как:

- карточка;
- строка;
- круговой показатель;
- график;
- панель без рамки;
- изображение;
- обложка;
- иной специализированный элемент.

### 12.4. Свободный холст

Произвольное пиксельное размещение, соединительные линии, стикеры и схемы реализуются отдельным типом содержимого или виджетом «Свободный холст».

Основной дашборд не должен становиться бесконтрольным пиксельным полотном.

## 13. Адаптивные раскладки

Каждое пространство поддерживает как минимум:

- большой экран;
- планшет;
- телефон.

По умолчанию система автоматически строит производные раскладки, сохраняя порядок и приоритеты.

Пользователь может отдельно настроить:

- порядок на телефоне;
- скрытие тяжёлых виджетов;
- компактное представление;
- замену широкого календаря списком;
- сворачивание дерева;
- отдельные размеры.

Изменения desktop-раскладки не должны молча уничтожать ручную мобильную настройку.

## 14. Пространства и верхняя панель

Каждая значимая верхняя вкладка открывает собственный холст.

На верхней панели непосредственно доступны:

- создание через `+`;
- перетаскивание и изменение порядка;
- переименование;
- выбор иконки;
- выбор цвета;
- закрепление;
- скрытие;
- дублирование;
- архивирование;
- привязка к папке, сфере или проекту;
- создание из шаблона.

Необходимо строго различать:

- удалить пространство или вкладку;
- удалить связанную папку;
- удалить сферу;
- удалить проект.

Удаление пространства по умолчанию удаляет только представление и раскладку. Предметные данные остаются.

## 15. Привязки пространств

### 15.1. Пространство папки

Контекстом становятся документы выбранной папки и её поддерева.

### 15.2. Пространство сферы жизни

Контекстом становятся объекты, связанные со сферой: проекты, задачи, события, документы, материалы и показатели.

### 15.3. Пространство проекта

Контекстом становятся проект и связанные с ним действия, документы, события, материалы и люди.

### 15.4. Пользовательское пространство

Не имеет обязательной предметной привязки. Каждый виджет настраивается отдельно.

Один объект может появляться во многих пространствах без копирования.

## 16. Виджеты и семантические связи

Документы, задачи, проекты, события, материалы, люди и другие предметные сущности могут иметь сохраняемые семантические связи.

Виджет обычно не является предметным объектом и не должен становиться полноценной конечной точкой доменных отношений.

Правило:

> Объекты связаны между собой; виджеты показывают объекты и их связи.

Исключением могут стать будущие сохраняемые представления, на которые пользователь сознательно ссылается как на отдельный ресурс. Такое решение требует отдельной архитектурной фиксации.

## 17. Переиспользование представлений в документах

Рендереры объектов и коллекций должны по возможности использоваться и на холсте, и при встраивании внутрь документов.

Пример:

- карточка задачи на дашборде;
- та же задача, встроенная в документ;
- та же задача в карточке проекта.

Это разные контексты отображения, но одна система компонентов и одна доменная команда изменения.

Такой подход нужен для будущих живых встраиваний:

- документов;
- задач;
- проектов;
- событий;
- материалов;
- списков;
- календарных представлений;
- связанных объектов.

## 18. Безопасность данных и раскладок

Обязательные правила:

1. Удаление виджета не удаляет источник данных.
2. Удаление пространства не удаляет связанные объекты.
3. Изменение раскладки не изменяет предметные отношения.
4. Конфигурации виджетов версионируются и мигрируются.
5. Неизвестная или повреждённая конфигурация изолируется и не ломает всё пространство.
6. Перед несовместимой миграцией создаётся резервная копия раскладок.
7. Должны существовать undo/redo и история значимых изменений компоновки.
8. Автосохранение раскладки не должно блокировать взаимодействие.
9. Обновление приложения не должно сбрасывать пользовательские пространства к значениям по умолчанию.
10. Шаблон и созданное из него пространство после создания являются независимыми конфигурациями.

## 19. Расширяемость

Система должна позволять добавлять новые виджеты без изменения центрального компонента каждого пространства.

Новый виджет регистрирует:

- тип;
- версию;
- семейство;
- источники;
- представления;
- настройки;
- действия;
- ограничения раскладки;
- миграции;
- renderer;
- editor/inspector.

На первом этапе не допускается произвольное выполнение непроверенного пользовательского JavaScript внутри виджетов.

Расширения и API проектируются позже через отдельную безопасную модель разрешений.

## 20. Начальный обязательный набор

Первая полноценная версия новой системы должна поддерживать минимум:

1. документ;
2. список задач;
3. одну задачу;
4. проект;
5. ближайшие события;
6. календарь;
7. дерево документов;
8. папку документов;
9. материалы;
10. связанные объекты;
11. обратные ссылки;
12. быстрый ввод;
13. показатель;
14. произвольный текст;
15. изображение;
16. ссылка;
17. контейнер-секцию.

Существующие «Обзор дня», «Главный фокус», «План дня», «Входящие», «Погода» и «Рекомендации» должны быть перенесены в новую модель, а не существовать отдельной несовместимой системой.

## 21. Шаблоны

Пользователь может начать с готового пространства и затем полностью его изменить.

Начальные шаблоны:

- Главная;
- сфера жизни;
- проект;
- обучение;
- чтение и материалы;
- здоровье;
- пустой холст.

Шаблоны не должны ограничивать дальнейшее редактирование.

Принцип постепенного раскрытия:

- сначала хороший готовый результат;
- затем простая настройка;
- затем глубокая настройка источников и представлений.

## 22. Не входит в первый этап реализации

- произвольный пользовательский код;
- бесконечная вложенность контейнеров;
- полноценное совместное редактирование холста;
- CRDT;
- сложные автоматизации;
- маркетплейс виджетов;
- бесконтрольное перекрытие карточек;
- обязательная ручная настройка мобильной версии;
- перенос всех доменных сущностей в один универсальный физический тип;
- удаление существующих данных при переходе на новую систему.

## 24. Критерии первого пользовательского среза

Первый законченный срез считается готовым, когда пользователь может:

1. создать верхнюю вкладку прямо на панели;
2. назвать её, выбрать иконку и цвет;
3. открыть пустой холст;
4. нажать `+` в нужном месте;
5. добавить документ, список задач и календарь;
6. перетащить каждый виджет;
7. свободно изменить ширину и высоту;
8. настроить источник данных рядом с виджетом;
9. дублировать и удалить виджет;
10. убедиться, что удаление виджета не удалило данные;
11. выйти из режима компоновки и нормально работать с содержимым;
12. обновить страницу и увидеть ту же раскладку;
13. открыть пространство на телефоне и получить пригодную автоматическую компоновку;
14. изменить порядок вкладок прямо на верхней панели;
15. удалить пространство без удаления связанной папки, сферы или проекта.

## 25. Архитектурные запреты

Запрещается:

- превращать виджет в владельца копии предметных данных;
- прошивать каждый новый виджет непосредственно в огромный центральный компонент;
- хранить конфигурацию без версии и проверки;
- связывать продуктовую модель с четырьмя фиксированными размерами;
- смешивать обычное взаимодействие с данными и случайное перемещение карточек;
- удалять данные при удалении представления;
- делать все верхние вкладки одним и тем же глобальным дашбордом;
- заставлять пользователя уходить в глубокие настройки для базовой компоновки;
- строить desktop-only холст без стратегии телефона;
- объявлять сетку готовой только потому, что drag-and-drop технически работает.

## 26. Открытые решения перед кодом

Перед реализацией необходимо отдельно выбрать и проверить прототипами:

- конкретный layout engine или собственный движок;
- число логических делений сетки;
- правила автоматического уплотнения;
- модель контейнеров;
- хранение responsive overrides;
- формат истории раскладок;
- взаимодействие drag-and-drop с редакторами текста и календарём;
- доступность с клавиатуры;
- производительность при большом числе виджетов;
- стратегия виртуализации длинных коллекций;
- правила переноса старых `DashboardWidget` в новую модель.

Выбор библиотеки не должен предшествовать утверждению этой модели и не должен менять продуктовый смысл системы.

## 27. Итоговое решение

ПСОЖ строит не набор заранее заданных карточек и не одну неизменяемую Главную.

Целевая система представляет собой:

> набор самостоятельных пространств верхней панели, каждое из которых является глубоко, но удобно настраиваемым адаптивным холстом; холст собирается из типизированных виджетов, а виджеты показывают одни и те же связанные предметные объекты без копирования и владения ими.

Сетка, каталог виджетов, редактор документов и семантический граф являются разными слоями одной системы:

- граф отвечает за реальные связи объектов;
- доменные модули владеют данными;
- представления определяют способ показа;
- виджеты размещают представления на холсте;
- пространства объединяют виджеты в пользовательский рабочий контекст.
