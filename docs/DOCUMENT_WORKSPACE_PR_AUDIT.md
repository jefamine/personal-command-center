## 1. Назначение документа

## 2. Основание и приоритет источников

Аудит выполнен после сверки с:

1. `PROJECT_FOUNDATION_ADDENDUM.md`;
2. `PROJECT_FOUNDATION.md`;
3. `PRODUCT_MAP.md`;

При конфликте продуктовый фундамент имеет приоритет над локальным ADR и реализацией.

Неподвижные принципы:

- один предметный объект не должен иметь две независимо редактируемые канонические копии;

- интерфейс не должен самостоятельно выбирать физический источник данных;

- миграции должны быть обратимыми или предваряться проверяемой резервной копией;
- работающий механизм не равен пользовательскому или продуктовому срезу.

## 4. Что фактически реализовано качественно

### 4.1. Два режима и единая граница

Контракт предусматривает:

- `internal`;
- `external-folder`;
- общий асинхронный `DocumentStorageAdapter`;
- типизированные команды и ошибки;
- отсутствие зависимости доменного контракта от React и конкретного browser API.

Это соответствует утверждённому гибридному направлению архитектуры.

### 4.2. Канонический внешний Markdown без скрытого зеркала body

`StoredDocument` является свежей проекцией физического файла. В комментариях контракта прямо запрещено сохранять его body как кэш в `DashboardState`.

Отдельная browser persistence хранит:

- descriptor;
- directory handle;
- metadata index;
- пути;
- хеши;
- ID;
- теги и aliases;
- состояние и ошибки scan.

В persisted index отсутствуют:

- body;
- raw frontmatter;
- полная копия Markdown.

Это соответствует принципу единственного канонического текста.

### 4.3. Read-only scan

Scan:

- рекурсивно находит Markdown;
- игнорирует `.obsidian`, `.psozh`, `.git` и временные файлы;
- не создаёт `.psozh`;
- не добавляет `psozh-id`;
- не меняет YAML;
- не прерывает весь scan из-за одного нечитаемого или повреждённого файла;
- сообщает duplicate ID и case-insensitive path collisions;
- оставляет конфликтующие документы раздельными.

Это важное и корректное соответствие ADR-002.

### 4.4. Managed и unmanaged identity

Реализовано различие:

- managed-файл: устойчивый `psozh-id`;
- unmanaged-файл: path-based identity;
- duplicate ID: отдельные conflict identities без объединения файлов;
- rename и move managed-файла сохраняют `DocumentId`.

Сканирование unmanaged-файла не модифицирует его. Действие make-managed является явной операцией.

### 4.5. Консервативная работа с frontmatter

Markdown parser хранит:

- BOM;
- исходные line endings;
- raw prefix;
- raw frontmatter;
- body offset;
- статус frontmatter;
- признаки безопасной metadata-правки.

Обычная замена body повторно использует исходный prefix без пересериализации.

Поддерживается минимальная правка:

- `psozh-id`;
- `tags`;
- `aliases`.

Неизвестные свойства, комментарии и форматирование остаются нетронутыми. Сложный или повреждённый YAML блокирует небезопасную metadata-правку, но не обязательно безопасную правку body.

### 4.6. Безопасные относительные пути

Проверяются:

- абсолютные пути;
- UNC;
- drive-qualified paths;
- выход через `..`;
- Windows forbidden characters;
- reserved names;
- trailing dots/spaces;
- control characters;
- длина сегмента и полного пути;
- case-insensitive collision keys.

### 4.7. Оптимистическая защита от внешней правки

Контракт и адаптеры используют `expectedContentHash`.

Перед изменяющими операциями выполняются повторные чтения и проверки идентичности/хеша. Stale reference возвращает typed `external-modification`, а не молча перезаписывает версию Obsidian или другой программы.

### 4.8. Verified copy-before-delete

Rename и move в браузерной модели реализованы как:

1. чтение исходника;
2. создание destination;
3. повторное чтение и проверка destination;
4. повторная проверка source;
5. удаление source;
6. чтение destination.

Аналогичный осторожный подход используется для папок и корзины.

### 4.9. Файловая корзина

Удаление из ПСОЖ:

- создаёт Markdown-копию в `.psozh/trash`;
- создаёт manifest;
- проверяет копию;
- повторно проверяет source;
- только затем удаляет исходник.

Restore проверяет manifest, hash и destination collision. Purge удаляет выбранный tombstone.

### 4.10. Изолируемый browser boundary

File System Access API изолирован в `BrowserDirectoryFilePort` и `BrowserDirectoryDocumentStorageAdapter`.

Основная policy находится в общем Markdown adapter, который тестируется через memory port. Это соответствует запрету прямого файлового API из React.

## 5. Что пока только заявлено или отсутствует

PR изменяет 26 файлов: документацию, доменные контракты, инфраструктурные адаптеры и тесты. Он не содержит законченной прикладной и пользовательской интеграции.

### 5.1. Нет workspace session

Не реализован единый runtime-координатор, который:

- загружает descriptor;
- восстанавливает handle;
- проверяет permission;
- открывает adapter;
- запускает scan;
- сохраняет index;
- предоставляет UI каталог и команды;
- переключает internal / external mode;
- обрабатывает disconnect/reconnect.

### 5.2. Нет интеграции с существующим DocumentRepository и UI

`InternalDocumentStorageAdapter` существует, но общий storage layer ещё не подключён как реальная единая граница для:

- `WorkspaceView`;
- `DocumentWidget`;
- object document view;
- создания и редактирования документов;
- удаления и восстановления.

UI пока не доказал, что он перестал различать физические источники.

### 5.3. Нет выбора папки в пользовательском сценарии

Есть низкоуровневые функции:

- feature detection;
- directory picker;
- query permission;
- request permission.

Нет пользовательского flow:

- явный выбор режима;
- read-only preview scan;
- сводка перед активацией;
- предупреждения;
- повторный выбор папки;
- отказ в разрешении;
- неподдерживаемый браузер;
- переключение обратно на internal.

### 5.4. Нет дерева и документного браузера

В PR нет интерфейса:

- папок;
- вложенного дерева;
- выбора документа;
- создания внутри;
- перемещения drag-and-drop;
- отображения managed/unmanaged/conflict/malformed states;
- поиска и открытия файла.

Следовательно, «документный браузер» этим PR ещё не реализован.

### 5.5. Нет миграционного мастера

ADR-002 требует:

- dry-run;
- safety backup;
- immutable plan;
- подтверждение;
- запись;
- повторное чтение и проверку;
- manifest;
- отдельную активацию external mode;
- rollback без удаления созданных Markdown-файлов.

В текущем PR этого процесса нет.

### 5.6. Нет пользовательского разрешения конфликтов

Есть typed conflict и `saveConflictCopy`, но отсутствует UI со сценариями:

- перечитать внешнюю версию;
- сравнить;
- сохранить локальный draft отдельной копией;
- отменить;
- объяснить последствия.

### 5.7. Нет обновления wiki-ссылок при rename/move

ADR-002 обещает обновлять только ранее однозначно разрешённые wiki-token и возвращать summary частичного результата.

Текущие `renameDocument` и `moveDocument` физически перемещают файл, но в этом PR нет интеграции с reference index и безопасного переписывания ссылок в source-файлах.

Это должно быть либо реализовано до объявления сценария завершённым, либо честно вынесено из текущего milestone и исправлено в ADR/ROADMAP.

### 5.8. Нет ручной проверки на настоящем vault

Автоматические тесты используют memory port и browser-like mocks. Не зафиксирована ручная проверка реального File System Access API на тестовой папке Windows/Chromium.

### 5.9. Нет независимого CI

На head PR нет GitHub status checks. Сообщения о локально зелёных тестах не подтверждены серверной CI.

## 6. Блокирующие технические дефекты

### P0-1. Невалидированный supplied ID в make-managed

При `createDocument` переданный ID проходит через `managedExternalDocumentId` и строгую проверку.

При `makeDocumentManaged` явно переданный `command.documentId` используется напрямую, тогда как только сгенерированный ID проходит через validator.

Риск:

- запись чрезмерно длинного или иного невалидного `psozh-id`;
- после записи файл может быть классифицирован как malformed;
- операция сама создаёт состояние, которое затем считает повреждённым.

Требование:

- любой desired ID должен проходить одну каноническую runtime-проверку до изменения файла;
- добавить regression tests для invalid supplied ID и отсутствия любых файловых изменений.

### P0-2. Rollback может удалить внешне изменённый destination

После создания destination ряд rollback-path удаляет его без `expectedContentHash`.

Если Obsidian или другая программа успела изменить destination между созданием и rollback, ПСОЖ может удалить уже внешне изменённый файл.

Риск:

- нарушение принципа «внешнее изменение не удаляется или не перезаписывается молча»;
- потеря данных при редкой, но реальной гонке.

Требование:

- все cleanup deletes созданных копий должны передавать ожидаемый hash;
- при несовпадении не удалять destination;
- возвращать `recoveryRequired: true` и `affectedPaths`;
- добавить adversarial tests для внешней правки destination непосредственно перед cleanup.

### P0-3. Неполная транзакционность trash restore/purge

Restore удаляет сначала trash Markdown, затем manifest. Purge делает то же.

При сбое второй операции остаётся осиротевший manifest. Это не обязательно теряет восстановленный пользовательский файл, но оставляет повреждённое служебное состояние.

Требование:

- определить идемпотентную модель tombstone lifecycle;
- сделать повторный restore/purge безопасным;
- добавить recovery/cleanup behavior для orphan manifest или orphan payload;
- покрыть отказ каждой отдельной операции тестами.

### P0-4. ADR и реализация расходятся по rename propagation

ADR помечен как `принято` и описывает обновление wiki-token, но код этого PR его не реализует.

Требование перед merge документации:

- либо реализовать обещанное поведение;
- либо явно маркировать его как следующий milestone, не выдавая за свойство текущей реализации.

## 7. Важные, но не обязательно блокирующие улучшения

### P1-1. Усилить runtime normalization browser persistence

В persistence используются несколько broad casts после ручных проверок:

- access state;
- external document state;
- scan error code;
- `Object.fromEntries(...) as unknown as DocumentScanSummary`;
- массивы после map/filter.

Это не главный риск данных, но слой, читающий сохранённое metadata, должен быть максимально строгим и без wholesale casts.

### P1-2. Общий plain-record helper

В нескольких файлах `isRecord` означает только non-null non-array object. Экземпляры классов и нестандартные prototypes могут пройти такую проверку.

Следует использовать единый `isPlainRecord` для данных, пришедших из JSON/IndexedDB, где это ожидается контрактом.

### P1-3. Явная семантика body update при malformed YAML

Реализация разрешает безопасную замену body, когда закрывающий delimiter найден, даже если metadata malformed/unsupported.

Это разумно и подтверждено тестами, но ADR формулирует главным образом блокировку metadata operations. Следует явно закрепить:

- body update разрешён, если body boundary однозначна;
- raw prefix сохраняется полностью;
- metadata patch блокируется.

### P1-4. Производительность scan

Некоторые операции перед созданием или make-managed выполняют полный scan для поиска duplicate ID. На большом vault это может быть дорого.

До первого рабочего среза допустимо, но session должна использовать актуальный индекс и ясную invalidation strategy, не превращая каждую команду в полный рекурсивный scan.

### P1-5. Поведение с non-Markdown файлами при операции папки

Browser adapter отказывается перемещать папку с non-Markdown файлами, чтобы не трогать вложения вслепую. Для текущего текстового milestone это безопасное ограничение.

UI обязан объяснять его заранее. В дальнейшем нужна отдельно спроектированная модель вложений и двоичных файлов, а не снятие проверки без нового контракта.

## 8. Что сохранить при продолжении работы

Не переписывать без доказанной причины:

- `DocumentStorageAdapter` и typed results;
- `DocumentWorkspaceFilePort`;
- разделение browser port и shared Markdown engine;
- read-only scan;
- metadata-only persistence index;
- managed/unmanaged/conflict identity model;
- conservative Markdown parser/mutator;
- path guards;
- expected hash checks;
- verified copy-before-delete;
- `.psozh` isolation;
- memory adapter и adversarial test approach;
- внутренний adapter как disposable projection над существующим repository.

1. Исправить P0-1 — validation supplied ID.
2. Исправить P0-2 — hash-guarded cleanup и recoveryRequired.
3. Исправить P0-3 — идемпотентность trash restore/purge и тесты отказов.
4. Устранить расхождение ADR с фактическим rename propagation.
5. Добавить workspace session/application coordinator.
6. Подключить internal и external adapters через единый прикладной интерфейс.
7. Реализовать пользовательский flow выбора папки и reconnect permission.
8. Реализовать минимальный документный браузер/дерево.

## 10. Рекомендуемое разделение дальнейшей работы

Чтобы не смешивать архитектуру, UI и миграцию в одном diff:

### Работа A. Hardening файлового ядра

- четыре P0-проблемы;
- дополнительные adversarial tests;
- без UI и миграции.

### Работа B. Workspace session и подключение

- descriptor/handle/index lifecycle;
- permission states;
- reconnect/disconnect;
- выбор adapter;
- без визуальной переделки редактора.

### Работа C. Первый пользовательский browser slice

- выбрать папку;
- read-only scan summary;
- увидеть дерево;
- открыть и отредактировать файл;
- создать документ и папку;
- rename/move/delete/restore;
- увидеть conflict/malformed/unmanaged states;
- перезагрузить страницу и восстановить подключение.

### Работа D. Миграционный мастер

- dry-run;
- backup;
- immutable plan;
- verified write;
- manifest;
- activation;
- rollback.

### Работа E. Wiki path resolution и rename propagation

- path-relative resolution;
- aliases;
- safe token rewriting;
- partial summary;
- отсутствие скрытых UUID bindings в Markdown.

## 11. Критерий принятия первого рабочего среза

Пользователь должен суметь без технических обходов:

1. выбрать internal или external-folder;
2. подключить папку явным действием;
3. увидеть read-only scan summary;
4. открыть дерево и документ;
5. отредактировать body без изменения стороннего frontmatter;
6. создать managed документ;
7. открыть unmanaged документ без автоматической модификации;
8. явно сделать его managed;
9. переименовать и переместить документ;
10. удалить его в файловую корзину и восстановить;
11. получить понятный конфликт при внешней правке;
12. сохранить локальный draft отдельной conflict copy;
13. перезагрузить приложение и восстановить подключение или получить честный reconnect flow;
14. переключиться обратно в internal mode без удаления внешних файлов;
15. убедиться, что IndexedDB не содержит вторую текущую копию внешнего body.

## 12. Итоговое решение

PR №1 не отклоняется и не переписывается с нуля.

Он принимается как ценный донор и технический фундамент файлового документного слоя, но не как завершённый документный workspace.

Следующее действие после возобновления Codex:

1. сначала hardening ядра;
2. затем workspace session и интеграция;
3. затем первый пользовательский browser slice;
4. только после этого merge как рабочая документная функция;
5. миграция внутренних документов — отдельный проверяемый этап с backup и rollback.
