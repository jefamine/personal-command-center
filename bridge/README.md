# Локальный мост Codex

Этот каталог позволяет дашборду и Codex обмениваться данными без OpenAI API.

## Снимок дашборда

Кнопка **«Опубликовать снимок»** создаёт `dashboard-snapshot.json` со схемой версии 2. Это не резервная копия всего приложения, а разрешённая проекция данных для Codex.

В снимок никогда не входят настройки, пути интеграций, журнал действий, личный контекст и память помощника. Записи раздела «Осмысление» передаются только через отдельную явно включённую категорию `journal`; по умолчанию она выключена. Связанные с ними заметки для Obsidian отсекаются независимо от настройки категории «Заметки». Для этого у них сохраняется внутренний признак происхождения, который не исчезает при удалении исходной записи или редактировании тега. Остальные обычные заметки передаются только при явно включённой категории. Служебные заметки задач, описания проектов и заметки событий отсекаются всегда. Отключённые категории представлены пустыми массивами. Опубликованный файл можно удалить отдельной кнопкой в интерфейсе.

Файл содержит:

- `schemaVersion` и время создания;
- точный `scope` из шести независимых категорий: задачи, проекты, календарь, заметки, осмысление (`journal`), материалы;
- только данные разрешённых категорий.

Если каталог проекта синхронизируется OneDrive или другим облачным сервисом, файлы моста тоже могут попасть в эту синхронизацию. Поэтому область снимка следует оставлять минимальной.

## Разбор записи

Кнопка разбора создаёт единственный активный `reflection-analysis-request.json` со схемой версии 3. В нём находятся исходный текст выбранной записи, её идентификаторы, политика анализа, только отмеченные в точном предпросмотре разделы личного контекста и только явно выбранные элементы памяти помощника. По умолчанию и личный контекст, и память пусты. Полный снимок дашборда к запросу не прикладывается.

Поле `request.memory` равно `null`, если пользователь ничего не выбрал, либо содержит снимок версии 1 с массивом `items`. Каждый элемент включает только `id`, точный `text` и `updatedAt`; состояние, источник и другие элементы памяти в файл не попадают. Сервер принимает от одного до шести уникальных элементов, не сокращает текст и отклоняет неизвестные поля, пустые строки, некорректные даты, текст длиннее 2500 символов в одном элементе и суммарный текст длиннее 12000 символов. Память передаётся только через этот одноразовый запрос и не добавляется в общий `dashboard-snapshot.json`.

Политика запроса помечает память как пользовательский контекст, а не как системные инструкции. Если старая формулировка расходится с текущей записью, приоритет имеет текущий текст; противоречие следует назвать или уточнить вопросом, а не скрыто разрешать в пользу памяти.

После сохранения результата дашборд оставляет в истории только `id` и версию выбранных формулировок. Второй скрытой копии текста там нет: неизменённая память разрешается из текущего локального хранилища, а изменённая или удалённая помечается без восстановления старого текста.

Codex читает запрос, использует `docs/SVP_FOUNDATION.md` как системное основание и записывает результат в `reflection-analysis-response.json` по формату из `reflection-analysis-response.example.json`. Контракт ответа остаётся версией 3 и сохраняет полный богатый объект `analysis`; отдельные `meaning`, `question` и `next_action` не являются полями файла моста.

Дашборд принимает ответ только когда одновременно совпадают:

- `requestId`;
- `entryId`;
- `sourceUpdatedAt` — запись не менялась после отправки.
- `requestDigest` — контрольная сумма точного текста, выбранного личного контекста и выбранного снимка памяти.

Контрольную сумму для ответа нужно дословно скопировать из `binding.requestDigest` активного запроса. Повторный POST с тем же идентификатором, но другим текстом, контекстом или набором памяти отклоняется. Запрос действует 24 часа.

Ответ остаётся черновиком: он не создаёт задачи и не меняет личный профиль автоматически. Только после фактического сохранения черновика в IndexedDB совпавшие request/response удаляются без архива чувствительного текста; сам черновик остаётся в локальном состоянии до решения пользователя. Ожидание можно отменить, оставив исходную запись только на устройстве.

Начиная с `DashboardState` v11 интерфейс только для новых ответов локально формирует необязательные предложения: `meaning` из `analysis.possibleExplanation`, `question` из `analysis.question`, `next_action` из `analysis.proposedAction`; текущая схема приложения — v13. Это не новый запрос к Codex и не изменение схемы bridge. Редактирование, принятие или отклонение каждого предложения не выполняет внешних действий. Перенос в локальную заметку либо создание незапланированной задачи во «Входящих» требуют отдельных кнопок; Obsidian обновляется только следующим ручным экспортом. Старые анализы v10 не получают решения ретроспективно.

Сферы жизни не добавлены в снимок bridge: `LifeArea[]`, их описания и `Project.areaId` остаются локальными. При включённой категории проектов снимок версии 2 содержит только прежнее строковое поле `project.area`.

Будущий технический долг v4 — связать точный ответ собственной контрольной суммой и сделать подтверждение очистки идемпотентным. В bridge v3 это ещё не реализовано.

## Команды для дашборда

Codex записывает массив действий в `codex-commands.json`. Пользователь открывает экран **«Интеграции»**, нажимает **«Проверить команды»**, просматривает очередь и явно применяет разрешённые действия. Применённый пакет переносится в `archive/`.

Поддерживаемые типы:

- `add_task` — поле `payload` содержит обязательный непустой `title` и только поля новой задачи; `done`, `generatedFromTaskId`, служебные и произвольные поля отклоняются;
- `update_task` — `entityId` и только изменяемые поля `title`, `notes`, `status`, `projectId`, `priority`, `estimateMinutes`, `energy`, `context`, `dueDate`, `scheduledDate`, `recurrence` в непустом `payload`. Служебные `id`, `createdAt`, `updatedAt`, `completedAt`, `generatedFromTaskId` и произвольные поля отклоняются. Значение статуса `done` здесь запрещено;
- `complete_task` — отдельная команда с `entityId`, отдельным пользовательским разрешением и без `payload`;
- `add_note` — поле `payload` содержит обязательный непустой `title` и только `body`, `projectId`, `tags`, `pinned`;
- `update_note` — `entityId` и только редактируемые поля `title`, `body`, `projectId`, `tags`, `pinned` в `payload`. Служебные `id`, `createdAt`, `updatedAt` и приватный признак `origin` команда менять не может.
- `add_reading` — ссылка, подборка или статья для виджета «Материалы для вас»; внешняя ссылка может использовать только `http` или `https`.

Все конверты команд и их `payload` проверяются по точному списку полей, типам и пределам размера. Некорректная команда не попадает в очередь интерфейса.

Для начала можно скопировать содержимое `codex-commands.example.json` в `codex-commands.json` и заменить пример своими данными.
